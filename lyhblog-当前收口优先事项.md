你的项目已经不是“跑不起来”的状态了。

现在的真实情况是：

- 后端能启动

## 优先级 1：把 AI 助手从“能跑”变成“稳定可展示”

### 为什么最值

这是你当前项目里最能拉开差距的点。

普通博客项目很多人都会做，但：

- 后台 AI 写作助手
- 生成标题、摘要、标签、提纲、正文草稿
- 一键回填表单

这套链路是可以直接写进简历的。

### 你现在最该补的点

#### 1. AI 接口超时策略

你现在已经确认：

- 普通接口没问题
- AI 接口大概要几十秒
- 前端之前 10 秒超时会把请求提前掐掉

你已经单独在 `src/api/ai.js` 里给 AI 请求加了更长超时，这个思路是对的。现在要做的是：

- 保证只有 AI 接口走长超时
- 其他接口仍保持普通超时

#### 2. AI 网关报错友好化

你已经遇到过：

- SSL 连接中断
- 上游限流
- 浏览器超时

所以现在最该补的是：

- 后端把 `Gateway request failed` / `rate_limit_error` 这类错误转成更清晰的中文提示
- 前端弹窗提示不要直接把底层英文甩给用户

建议前端最终提示风格：

- `AI 服务当前较忙，请稍后重试`
- `当前模型请求过多，请稍后再试`
- `AI 网关连接异常，请检查配置或稍后重试`

#### 3. AI 返回 JSON 稳定性

你现在的后端逻辑是：

- Claude 返回文本
- 后端直接 `json.loads(output_text)`

这在演示阶段能用，但有风险：

- 只要模型多吐一句解释
- 或包一层 ```json 代码块
- 或返回不完整 JSON

整个接口就会 502。

你接下来值得补的是：

- 先做最小清洗
- 再做 `json.loads`

例如后面可以考虑在 service 里：

- 去掉 markdown 代码块包裹
- 截取第一个 `{` 到最后一个 `}`
- 再做 JSON 解析

这一步不是今天必须改，但它是当前 AI 功能里最值得补的稳定性点。

### 验收标准

你应该能稳定做到：

1. 打开后台新增文章页
2. 点击 `AI 助手`
3. 输入主题
4. 返回结果
5. 一键回填
6. 保存文章

至少连续成功 2 到 3 次，才算真正稳定。

## 优先级 2：把时间、配置、环境变量这些基础项彻底收干净

### 为什么最值

这些东西看起来不炫，但如果不收口：

- 你自己换终端会跑不起来
- WSL / Ubuntu 部署会出问题
- 简历项目别人拿到也复现不了

### 你现在最该做的

#### 1. 时区改成中国时间

你刚确认过：

- 当前项目还是 `UTC`

你应该把 Django 配置改成：

```python
TIME_ZONE = 'Asia/Shanghai'
USE_TZ = True
```

这一步很值，因为它会直接影响文章创建时间的展示。

#### 2. 明确运行时到底读哪个文件

你现在已经搞清楚了：

- 运行时读 `.env`
- `.env.example` 只是模板

所以你要把这件事彻底固定下来：

后端：

- `lyhblog-backend/.env`：真实配置
- `lyhblog-backend/.env.example`：模板

前端：

- `lyhblog-frontend/.env.development`
- `lyhblog-frontend/.env.example`

#### 3. 清掉混用配置

你现在 `settings.py` 里同时留着：

- `OPENAI_*`
- `ANTHROPIC_*`

如果你已经确定这版就走 Claude 中转，那后面最好收口成一套主线，避免自己未来看晕。

不是说现在一定要立刻删，而是：

- README 里要写清楚当前主线
- `.env.example` 里别混得太乱

### 验收标准

你应该能做到：

- 新开一个终端也能直接启动后端
- `.env` 不需要临时手动 export 变量
- 时间显示是中国时间

## 优先级 3：把前台/后台文案和乱码彻底收一遍

### 为什么最值

你现在项目最大的问题之一，不是功能少，而是：

**成品感不够统一。**

特别是：

- 部分中文乱码
- 按钮文案不统一
- AI 助手文案不够自然
- 分类和标签结果有时会很怪

如果这些不收，项目虽然能跑，但看起来会很像练手项目。

### 你现在最该修的点

#### 1. 后台文章编辑页

重点看：

- 标题
- 摘要
- 正文
- AI 助手按钮
- 成功 / 失败提示

#### 2. AI 助手抽屉

重点看：

- 字段名是否自然
- placeholder 是否统一
- 成功提示是否自然
- “应用全部 / 应用到正文 / 应用已匹配标签”这些文案是否顺

#### 3. 后端 message

比如：

- `AI 建议生成成功`
- `AI 服务当前较忙，请稍后重试`
- `请输入主题，或提供现有草稿内容`

这些都值得统一成正常中文。

### 验收标准

你打开后台时，不应该再看到：

- 奇怪乱码
- 提示语像机器拼出来的
- 用户看不懂的英文底层错误

## 优先级 5：部署只做“速通能跑”，先别追求完美

### 为什么这样定

你现在已经不是完全没部署基础，但也还没到做完整 DevOps 的阶段。

所以你现在上线的目标应该是：

**先跑通，再优化。**

### 当前最现实路线

1. 本地 Windows 跑通
2. WSL / Ubuntu 跑通
3. Gunicorn + Nginx 跑通
4. 前端 `npm build` 后交给 Nginx
5. 数据库连接正常
6. AI 中转配置正常

### 现在先不要纠结的事

- Docker 一上来全套化
- 多环境特别复杂的编排
- 监控、日志系统、灰度发布

这些以后再说。

### 验收标准

你要至少做到：

- Linux 环境里前后端能打开
- 后台能登录
- AI 助手能跑一次

这就够你写“有部署经验”了。
